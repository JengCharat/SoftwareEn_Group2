โปรเจกต์นี้เป็นส่วนหนึ่งของรายวิชา **CP353004 Software Engineering**  
วัตถุประสงค์ของเอกสารนี้คือสรุปภาพรวมระบบเดิม, ผู้มีส่วนเกี่ยวข้อง (Stakeholders), Requirement, อธิบาย **Activity Diagram (1 Flow)** และจัดทำ **Customer Journey Map (Stakeholder 1 ประเภท)** พร้อม Action Items เพื่อนำไปต่อยอดออกแบบฟังก์ชันของระบบ

---

## 1) Project Overview (ภาพรวมระบบเดิม)

**ไปนำแหน่** เป็นระบบที่ช่วยเชื่อมต่อการเดินทางแบบ “ร่วมทาง/คาร์พูล” ระหว่างผู้โดยสารและผู้ขับขี่  
เป้าหมายคือช่วยให้ผู้โดยสารสามารถค้นหาเส้นทางที่ตรงกับความต้องการ ส่งคำขอร่วมเดินทาง และรอการตอบรับจากผู้ขับขี่ได้อย่างเป็นระบบ

---

## 2) Stakeholders (ผู้มีส่วนเกี่ยวข้อง)

ระบบมีผู้ใช้งานหลัก 4 กลุ่ม

1. **Guest**: ผู้ใช้งานทั่วไปที่ยังไม่เข้าสู่ระบบ (ดูข้อมูลพื้นฐานได้บางส่วน)
2. **Passenger**: ผู้โดยสาร (ค้นหาเส้นทาง, ส่งคำขอร่วมเดินทาง, ติดตามสถานะ)
3. **Driver**: ผู้ขับขี่ (สร้างเส้นทาง, ตอบรับ/ปฏิเสธคำขอร่วมเดินทาง)
4. **Admin**: ผู้ดูแลระบบ (จัดการข้อมูล/ดูแลความปลอดภัย/ตรวจสอบผู้ใช้)

---

## 3) Requirements (ความต้องการของระบบ)

### Functional Requirements (ตัวอย่างที่เกี่ยวกับ Flow นี้)

- ผู้โดยสารสามารถ **ค้นหาและจับคู่เส้นทาง** จากต้นทาง/ปลายทาง/เวลา/จำนวนที่นั่ง
- ผู้โดยสารสามารถ **ส่งคำขอร่วมเดินทาง (Request Booking)**
- ระบบต้องตรวจสอบเงื่อนไข เช่น **ที่นั่งว่าง**
- ผู้ขับขี่สามารถ **ตอบรับ (Accept) / ปฏิเสธ (Reject)** คำขอ
- ระบบต้อง **อัปเดตสถานะการจอง** และแจ้งผลให้ผู้โดยสาร

### Non-Functional Requirements (เน้นที่สำคัญ)

- ความปลอดภัยและความน่าเชื่อถือ (เช่น โปรไฟล์คนขับ, การยืนยันตัวตน)
- ความชัดเจนของสถานะ (Pending / Accepted / Rejected)
- ความง่ายในการใช้งาน (ค้นหาไว, ข้อมูลอ่านง่าย)

---

## 4) Activity Diagram (1 Flow)

### Flow ที่เลือก: **Passenger Booking Flow (ผู้โดยสารค้นหาและจองร่วมเดินทาง)**

**เหตุผลที่เลือก Flow นี้:**  
เป็น Flow หลักของระบบที่สะท้อนประสบการณ์ผู้โดยสารโดยตรง ตั้งแต่ค้นหาเส้นทางไปจนถึงได้รับผลตอบรับจากผู้ขับขี่

### สรุปขั้นตอนหลัก

1. Passenger เปิดแอปและตรวจสอบการ Login
2. Passenger กรอกเงื่อนไขค้นหาและกดค้นหา
3. System จับคู่เส้นทาง และแสดงผลลัพธ์
4. Passenger เลือกเส้นทางและส่งคำขอ (Booking)
5. System ตรวจสอบที่นั่งว่างและสร้างสถานะ **Pending**
6. Driver ตอบรับ/ปฏิเสธคำขอ
7. System อัปเดตสถานะและแจ้ง Passenger

## 5) Customer Journey Map (Stakeholder 1 ประเภท)

### Stakeholder ที่เลือก: **Passenger (ผู้โดยสาร)**

#### Stages (ขั้นตอนการใช้งาน)

1. **ค้นหาเส้นทาง**
2. **พิจารณาและเลือกคนขับ**
3. **ขอร่วมเดินทาง (Booking)**
4. **รอการตอบรับ**
5. **เดินทาง/ติดตามรายละเอียด**

#### Pain Points (ปัญหาที่พบ)

- ข้อมูลคนขับไม่ชัดเจน ทำให้ไม่มั่นใจ
- ไม่พบเส้นทางที่ตรงเงื่อนไขบ่อย
- รอการตอบรับนาน ไม่รู้สถานะชัดเจน
- ถ้าโดนปฏิเสธ ต้องเริ่มหาใหม่ ทำให้เสียเวลา

#### Opportunities (โอกาสในการพัฒนา)

- เพิ่มการยืนยันตัวตน / Badge
- เพิ่มรายละเอียดโปรไฟล์คนขับ + รีวิว
- เพิ่มหน้า Booking Status และ Notification ที่ชัดเจน
- เพิ่มระบบ Timeout/คาดการณ์เวลาตอบรับ
